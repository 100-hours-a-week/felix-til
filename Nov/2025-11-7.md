### 🚀 Today I Learned
#### 락을 사용하는 이유
```
멀티스레드 환경에서는 여러 스레드가 프로세스의 공유자원, 즉 데이터(Heap/Data 영역)에 동시에 접근하여 값을 변경 할 수 있음

이때, 경쟁 상태(Race Condition)가 발생하여 데이터의 일관성(Consistency)이 깨지거나 예상치 못한 결과가 발생하는 것을 방지하기 위해 동기화(Synchronization) 매커니즘 필요

락(Lock)은 공유 자원에 접근하는 코드 영역(Critical Section)을 보호, 한 시점에 오직 하나의 스레드만 접근하고 수정할 수 있도록 보장할 수 있다.
```
#### 락 경쟁 병목 완화 전략
```
🎯 락 경쟁: 락을 획득하기 위해 대기하는 스레드가 많아져서 CPU가 대기하는 스레드를 스케줄링 하는 데 시간을 낭비하게 되어 병목 현상 발생.

🔐 락 분할 조정
하나의 큰 락으로 여러 독립적인 공유 자원을 보호하는 대신, 각 자원별로 독립적인 락을 사용하여 락의 범위를 좁히고, 동시에 여러 스레드가 각각 다른 락을 획득하여 병렬로 작업을 수행할 수 있도록 함 -> ConcurrentHashMap

🔐 락 프리 알고리즘
락을 전혀 사용하지 않거나, 스레드가 다른 스레드를 무한정 기다리지 않도록 보장하는 알고리즘 사용. -> CAS(Compare And Swap) 같은 하드웨어 원자적 연산 활용
-> CAS

🔐 읽기-쓰기 분리 및 R/W 락 사용
데이터의 읽기 빈도가 쓰기 빈도보다 월등히 높은 경우, 리드-라이트 락을 사용하여 다수의 읽기 작업을 병렬로 허용하고 쓰기 작업만 배타적으로 처리
```
