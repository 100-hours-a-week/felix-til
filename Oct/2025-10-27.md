### 🚀 Today I Learned
#### JWT 토큰
```
인증에 필요한 정보(사용자 ID 등)를 JSON 객체로 담아 비밀 키로 서명(Sign)한 후, 이 서명된 토큰 자체를 클라이언트에게 전달하는 무상태(Stateless) 기반의 인증 방식입니다.

사용자 로그인 성공 시, 서버는 사용자 정보를 담은 JWT를 생성하고 비밀 키로 서명합니다.
서버는 이 JWT를 클라이언트에게 전송하며, 클라이언트는 이를 로컬 스토리지나 쿠키에 보관합니다.
클라이언트는 이후 요청 시 JWT를 HTTP Authorization 헤더에 담아 서버에 전송합니다.
서버는 별도의 DB 조회를 하지 않고, 토큰의 서명(Signature)을 검증하여 위변조 여부와 유효성을 확인하고, 토큰 내부의 정보를 사용해 인증을 처리합니다.

구조: Header.Payload.Signature (헤더와 페이로드는 인코딩되어 누구나 볼 수 있습니다.)
저장 위치: 클라이언트 (로컬 스토리지, 세션 스토리지, 또는 HttpOnly 쿠키).

장점 (확장성/효율성): 서버가 별도로 세션을 관리할 필요가 없어 **무상태(Stateless)**가 되어 서버 확장성이 매우 뛰어나고 부하가 적습니다.
단점 (취약성): 
토큰 길이: 토큰 자체의 길이가 길어 요청 시마다 네트워크 부하가 세션 방식보다 클 수 있습니다.
정보 노출: Payload는 인코딩만 될 뿐 암호화되지 않으므로, 민감한 정보는 절대 담을 수 없습니다.
강제 로그아웃 어려움: 한 번 발급된 JWT는 만료되기 전까지 유효하므로, 토큰이 탈취되면 서버에서 이를 강제로 무효화하기 어렵습니다. (이를 극복하기 위해 Refresh Token 등을 사용합니다.)
```
#### 세션
```
정의 및 역할
사용자의 인증 정보를 서버 측에 저장 및 관리하는 방식입니다.

작동 방식:

사용자가 로그인에 성공하면, 서버는 사용자 정보를 기반으로 고유한 세션 ID를 생성하고 이를 서버의 메모리 또는 세션 저장소(DB, Redis 등)에 저장합니다.

서버는 이 세션 ID를 쿠키에 담아 클라이언트(브라우저)에게 응답합니다.

클라이언트는 이후 요청 시 해당 세션 ID가 담긴 쿠키를 서버에 자동으로 전송합니다.

서버는 받은 세션 ID로 서버 저장소를 조회하여 사용자를 식별하고 인증 상태를 확인합니다.

저장 위치: 서버 (Session DB, Redis, 서버 메모리 등).

장점 (보안): 민감한 사용자 정보가 서버에만 있으므로, 쿠키가 탈취되어도 세션 ID 자체는 무의미합니다. (단, 세션 ID 탈취를 통한 위장 가능성은 여전히 존재)
단점 (확장성/부하): 모든 세션 정보를 서버에서 관리해야 하므로, 동시 접속자가 많아지면 서버에 부하가 심해지고, 서버를 여러 대로 확장(Scale-out)할 때 세션 동기화 문제가 발생합니다.
```
